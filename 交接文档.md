## 核心架构说明

### 双模式设计
`app.py` 中的 `AppState` 类管理应用状态，有两种模式：

| 模式 | 触发条件 | 数据来源 |
|------|---------|---------|
| `mock` | 启动时无教材 | `src/mock_data.py`（硬编码假数据） |
| `real` | 上传文件或自动加载完成 | `src/` 中的真实逻辑模块 |

**启动时自动加载教材**：应用启动时 `auto_load_textbooks()` 会自动扫描 `C:\Users\user\Desktop\textbooks\` 目录，加载所有 PDF（排除含"赛题"的文件），解析并构建图谱和 RAG 索引。

### 后端真实逻辑链路

```
用户上传文件
    ↓
pdf_parser.py  →  按章节分块提取文本
    ↓
knowledge_extractor.py  →  MiniMax M2.7 提取知识点（每章节一次 LLM 调用）
    ↓
graph_builder.py  →  汇总所有教材节点和边，分配颜色
    ↓
integrator.py  →  三级对齐算法整合：
    ├── L1: 精确名称匹配
    ├── L2: BGE-small-zh Embedding 余弦相似度 > 0.85
    └── L3: MiniMax LLM 精判
    ↓
rag_pipeline.py  →  FAISS 索引，RAG 问答带引用溯源
```

### MiniMax API 配置（config.py）

```python
APP_LLM_BASE_URL = "https://api.minimaxi.com/anthropic"
APP_LLM_API_KEY  = "sk-cp-..."        # 从 .env 读取
APP_LLM_MODEL    = "MiniMax-M2.7"     # 必须使用 MiniMax-M2.7
```

API 通过 `src/llm_client.py` 的 `Anthropic` SDK 调用（兼容格式）。

### Embedding 与相似度

```python
EMBEDDING_MODEL = "BAAI/bge-small-zh-v1.5"
EMBEDDING_SIMILARITY_THRESHOLD = 0.85  # L2 粗筛阈值
```

## 当前已知问题

### 1. 启动时自动加载慢
`auto_load_textbooks()` 对每本教材调用 LLM 提取知识点，每本教材约 8 次 LLM 调用（`knowledge_extractor.py` 中 `MAX_LLM_CALLS_PER_BOOK = 8`）。7 本教材 × 8 次 × 平均 10 秒/次 ≈ **约 10 分钟**才能完成启动时的预加载。

**如需快速测试**，可以将 `knowledge_extractor.py` 中 `MAX_LLM_CALLS_PER_BOOK` 改为 2，或者暂时注释掉 `app.py` 末尾的 `auto_load_textbooks()` 调用。

### 2. PDF 解析章节颗粒度问题
`src/pdf_parser.py` 的 `parse_pdf()` 依赖"第X章"作为章节边界检测。部分 PDF 的目录页存在大量噪音章节（如"第五节 颈外侧区"），被误识别为独立章节。已在代码中做了合并处理，但仍需观察效果。

### 3. 压缩比计算方式
目前 `graph_builder.compute_compression_stats()` 只统计知识点 definition 文本的字数，而非"原始教材总字数 vs 整合后总字数"。这个指标偏保守。
