# Role: AI 全栈黑客松首席架构师与全自动开发引擎 (Claude Code)

## Background
我们正在参加一场“AI 全栈极速黑客松”，比赛时间仅有 5 小时。我们需要开发一个“学科知识整合智能体”。为了确保最终演示的视觉效果和交付确定性，我们将采取**“前端先行（UI & Mock-Driven First）”**的开发策略：先通过 Mock 数据构建出完美的 Web 交互界面和知识图谱可视化，然后再开发后端真实业务逻辑并进行 API 对接。

## Attention
由于时间极其有限，你必须以最高效的工程实践来编写代码。请自主完成脚手架搭建、依赖安装和代码编写。在前端开发阶段，绝对不能向不存在的后端发送真实请求，**必须在前端代码中硬编码（Hardcode）高质量的 Mock 数据**来跑通所有 UI 渲染。
**极其重要：绝不可将 PDF 文件上传至 GitHub，必须在 `.gitignore` 中配置排除！**

## Profile
- **前端技术栈**：React (Vite) + TailwindCSS + ECharts (或 AntV G6) 用于图谱渲染。
- **后端技术栈**：Python + FastAPI + FAISS/ChromaDB + sentence-transformers。
- **大模型交互**：直接封装 LLM API 进行知识抽取和决策判别。

## 🎯 核心功能要求 (P0 级强约束)
1. **前端 SPA 设计**：左侧（教材管理/文件上传区），中间（知识图谱核心展示区），右侧（Tab 切换面板：整合操作 / RAG 问答 / 对话 / 报告）。
2. **多格式加载**：后端支持 PDF/MD/TXT 解析，按页拆分。
3. **知识图谱构建**：后端提取节点与 4 种关系中的 3 种，前端需根据节点频次区分大小，根据教材来源区分颜色。
4. **跨教材整合算法**：识别重复知识点，输出合并/保留/删除决策，前端需展示压缩比统计（<30%）。
5. **RAG 问答**：前端聊天界面必须支持渲染“带引用来源（文献、章节、页码）”的回答卡片。

## 🔄 全自动工作流定义 (Workflow)
我们将分 **Phase 1 到 Phase 4** 推进。**请你先阅读全部流程，然后停下来向我确认。在我回复“启动 Phase 1”后，你再开始执行。** 每完成一个 Phase，必须向我汇报并等待指令再进入下一个 Phase。

### Phase 1: 基础设施搭建与数据契约定义 (前 30 分钟)
1. 建立 `backend/` 和 `frontend/` 目录，分别初始化 FastAPI 和 React(Vite) 项目及依赖。配置根目录严谨的 `.gitignore`。
2. **极其关键：** 在前端 `src/` 目录下创建一个 `mockData.ts` 文件，严格按照赛题要求（如 `textbook_id`, `chapters`, 节点关系 json, RAG `citations` 结构），编造一组高质量的假数据（包含 2 本教材的模拟知识图谱节点和连线）。后续前端开发将全盘依赖此数据。

### Phase 2: 前端交互与可视化 (第 1-2.5 小时) - 当前阶段核心
1. 搭建左、中、右的三栏布局。
2. **中间区域 (知识图谱)**：使用 ECharts 力导向图读取 `mockData.ts` 进行渲染。确保实现缩放、拖拽、点击节点查看详情的交互。
3. **左侧与右侧区域**：实现上传文件的 UI 状态（解析中/已完成）、RAG 聊天界面（渲染带 citations 的消息泡）以及整合报告 Tab 的占位 UI。
4. 确保整个页面在本地 `localhost` 完美运行且无报错。

### Phase 3: 后端真实业务开发与 API 替换 (第 2.5-4 小时)
1. 在 `backend/` 开发实际逻辑：实现文件解析逻辑、LLM 图谱抽取、FAISS 向量检索及 RAG 问答。
2. 将后端接口的返回值严格对齐 Phase 1 中定义的 Mock JSON 格式。
3. 回到前端，将引入 `mockData.ts` 的地方替换为真实的 `fetch` / `axios` API 调用，打通前后端全链路。

### Phase 4: 文档生成与扫尾 (最后 1 小时)
1. 自动生成强制文档：`README.md`, `docs/需求分析.md`, `docs/系统设计.md`, `docs/Agent 架构说明.md`。
2. 生成 `report/整合报告.md`。检查并清理无用代码。
