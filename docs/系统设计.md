# 系统设计文档

## 一、系统架构

本系统采用 **Gradio 单体应用架构**，所有功能集成在一个 Python 进程中，通过 Gradio Web UI 提供交互界面。

```
┌─────────────────────────────────────────────────────────────┐
│                    Gradio Web UI (port 7860)                 │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────────┐  │
│  │ 左栏      │  │ 中栏         │  │ 右栏                  │  │
│  │ 教材管理   │  │ 知识图谱     │  │ 整合/RAG/对话/报告    │  │
│  │ 文件上传   │  │ ECharts     │  │ Tab 切换面板          │  │
│  └──────────┘  └──────────────┘  └───────────────────────┘  │
└────────────────────────┬────────────────────────────────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
   ┌────────────┐ ┌────────────┐ ┌────────────┐
   │ PDF 解析    │ │ LLM 调用   │ │ 向量检索    │
   │ PyMuPDF    │ │ MiniMax    │ │ FAISS      │
   └────────────┘ └────────────┘ └────────────┘
          │              │              │
          ▼              ▼              ▼
   ┌────────────┐ ┌────────────┐ ┌────────────┐
   │ 知识点提取  │ │ 三级对齐    │ │ RAG 问答    │
   │ Extractor  │ │ Integrator │ │ Pipeline   │
   └────────────┘ └────────────┘ └────────────┘
```

### 双模式设计

`AppState` 类管理全局状态，支持两种运行模式：

| 模式 | 触发条件 | 数据来源 |
|------|---------|---------|
| Mock | 启动时无教材 | `src/mock_data.py`（硬编码假数据） |
| Real | 上传文件或自动加载完成 | `src/` 中的真实逻辑模块 |

启动时 `auto_load_textbooks()` 自动扫描教材目录，加载 PDF 并构建图谱和 RAG 索引。

## 二、核心数据流水线

```
用户上传 PDF/TXT/MD
        │
        ▼
  pdf_parser.py ─── 按章节分块提取文本（PyMuPDF，支持"第X章"识别）
        │
        ▼
  knowledge_extractor.py ─── MiniMax M2.7 提取知识点（每章节一次 LLM 调用）
        │
        ▼
  graph_builder.py ─── 汇总所有教材节点和边，分配颜色
        │
        ▼
  integrator.py ─── 三级对齐算法整合：
        │   ├── L1: 精确名称匹配（字符串完全一致）
        │   ├── L2: BGE-small-zh Embedding 余弦相似度 > 0.85
        │   └── L3: MiniMax LLM 精判（置信度 0.7-0.85）
        │
        ▼
  rag_pipeline.py ─── FAISS IndexFlatIP 索引，RAG 问答带引用溯源
        │
        ▼
  echarts_templates.py ─── 生成 ECharts HTML（力导向图）
```

## 三、技术选型理由

### 3.1 Web 框架：Gradio 5.x

| 考量 | 选择理由 |
|------|---------|
| 黑客松时间约束 | Gradio 纯 Python，无需前后端分离，开发效率最高 |
| 部署便捷性 | 魔搭创空间原生支持 Gradio，一条命令部署 |
| 交互能力 | 支持 Tab 切换、文件上传、实时图表（gr.HTML）、聊天界面 |
| 成熟度 | 医学 AI Demo 普遍采用 Gradio，社区支持丰富 |

**不选 Flask/FastAPI + React 的原因**：5 小时内无法完成前后端联调 + 图谱可视化 + 部署。

### 3.2 LLM：MiniMax M2.7

| 考量 | 选择理由 |
|------|---------|
| API 可用性 | 用户自有 MiniMax API Key，免费额度充足 |
| 中文能力 | MiniMax M2.7 中文理解和生成能力强，适合医学领域 |
| 兼容性 | OpenAI SDK 兼容格式，一行代码切换模型 |
| 速度 | 响应速度快（< 5 秒），适合黑客松快速迭代 |

### 3.3 向量检索：FAISS

| 考量 | 选择理由 |
|------|---------|
| 轻量性 | `faiss-cpu` 无需 GPU，pip install 即用 |
| 性能 | IndexFlatIP 精确搜索，万级向量毫秒响应 |
| 成熟度 | Meta 开源，RAG 领域标准工具 |
| 部署友好 | 无外部依赖（对比 ChromaDB/Pinecone） |

**不选 ChromaDB 的原因**：引入额外服务依赖，增加部署复杂度。

### 3.4 Embedding 模型：BAAI/bge-small-zh-v1.5

| 考量 | 选择理由 |
|------|---------|
| 中文优化 | 专为中文语料训练，医学文本语义捕捉能力强 |
| 模型体积 | small 版本仅 ~90MB，本地加载快 |
| 质量 | MTEB 中文榜单前列，512 token 上下文足够使用 |
| 本地运行 | 无需调用外部 API，避免网络延迟和费用 |

### 3.5 PDF 解析：PyMuPDF (fitz)

| 考量 | 选择理由 |
|------|---------|
| 速度 | C 扩展底层，解析速度远超 pdfplumber |
| 功能 | 支持文本提取、页面范围、目录识别 |
| 可靠性 | 医学教材 PDF 格式规范，PyMuPDF 处理效果好 |

### 3.6 图谱可视化：ECharts

| 考量 | 选择理由 |
|------|---------|
| 交互性 | 原生支持力导向图缩放、拖拽、节点高亮 |
| 美观度 | 专业级图表渲染，视觉效果远超 D3.js 默认样式 |
| 集成方式 | 通过 `gr.HTML` 嵌入 iframe，与 Gradio 无缝集成 |
| 数据驱动 | JSON 配置驱动，动态更新方便 |

## 四、模块职责

| 模块 | 职责 | 关键接口 |
|------|------|---------|
| `app.py` | Gradio 主应用，UI 构建，事件路由，全局状态管理 | `AppState`, `do_parse()`, `do_integrate()`, `rag_chat()` |
| `src/config.py` | 配置常量（API Key、阈值、路径） | 环境变量加载 |
| `src/llm_client.py` | LLM API 封装，支持 JSON 解析 | `call_llm()`, `call_llm_json()` |
| `src/pdf_parser.py` | PDF/TXT/MD 文件解析，章节拆分 | `parse_file()`, `parse_pdf()` |
| `src/knowledge_extractor.py` | LLM 驱动的知识点和关系提取 | `extract_knowledge_points()`, `process_textbook()` |
| `src/embedding.py` | BGE 模型加载和文本向量化 | `encode_texts()`, `find_similar_pairs()` |
| `src/graph_builder.py` | 知识图谱构建、颜色分配、压缩比计算 | `build_graph()`, `compute_compression_stats()` |
| `src/integrator.py` | 三级对齐整合算法（L1/L2/L3） | `run_integration()`, `level1_exact_match()` |
| `src/rag_pipeline.py` | RAG 问答管道（分块→索引→检索→生成） | `RAGPipeline.add_textbook()`, `search()`, `answer()` |
| `src/echarts_templates.py` | ECharts 力导向图 HTML 生成 | `render_knowledge_graph()` |
| `src/mock_data.py` | Mock 数据（UI 开发和演示用） | 18 节点、20 边、7 决策 |

## 五、数据模型

### 知识点节点

```json
{
  "id": "b01_n05",
  "name": "白细胞",
  "definition": "一类有核的血细胞...",
  "category": "核心概念",
  "chapter": "第三章 血液",
  "page": 78,
  "textbook": "生理学",
  "frequency": 4
}
```

### 关系边

```json
{
  "source": "b01_n05",
  "target": "b01_n09",
  "relation_type": "prerequisite",
  "description": "理解免疫应答需要先了解白细胞的功能"
}
```

### 关系类型（4 种）

| 类型 | 含义 | 示例 |
|------|------|------|
| `prerequisite` | A 是 B 的前置知识 | 静息电位 → 动作电位 |
| `parallel` | A 和 B 是并列关系 | 红细胞 ↔ 白细胞 |
| `contains` | A 包含 B | 血细胞 → 红细胞 |
| `applies_to` | A 应用于 B | 神经递质 → 突触传递 |

## 六、关键设计决策

### 6.1 为什么用三级对齐而不是纯 LLM？

- **效率**：L1 精确匹配 O(n) 解决 10-15% 的配对，无需 LLM 调用
- **成本**：L2 向量相似度本地计算，只将 L3 边界情况交给 LLM
- **可解释性**：每一级有明确的置信度分数，决策过程可追溯

### 6.2 为什么 RAG 用 500 字分块？

- 医学教材的知识单元通常在 **300-600 字**范围内
- 50 字重叠确保跨块边界的信息不丢失
- BGE-small-zh 最大 token 数 512，500 字在截断安全范围内

### 6.3 为什么单进程而不是微服务？

- 黑客松时间约束：5 小时无法完成服务拆分 + 网络通信 + 部署
- 数据量可控：7 本教材的知识图谱在内存中完全容纳
- Gradio 天然单进程，魔搭创空间也以单容器部署
